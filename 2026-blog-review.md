## 1. 강점 — 유지해야 할 것

### 1.1 트레이드오프 중심 서술
`2026-01-19-concurrency-comparison-test`, `2026-05-20-distributed-issue-limit-counter` 같은 글은 단순 "이렇게 했다"가 아니라 "A vs B vs C 중에 왜 B를 택했고, A/C를 왜 버렸는가"를 표로 정리한다. 면접에서 가장 잘 먹히는 글 형태다.

> 인용: "**겉보기 비용이 낮아도 분산에선 비쌀 수 있다.** O(1) 카운터의 실제 비용은 연산 자체보다 직렬화 + RAC 노드 간 블록 이동에서 나올 수 있다."  
> — distributed-issue-limit-counter

이런 한 줄 통찰이 글마다 살아있는 게 큰 자산이다.

### 1.2 실측 데이터와 관찰지표 결합
`concurrency-comparison-test`는 VU=100/500의 avg/p95/p99, CPU/Heap/GC pause, DB pending thread까지 들고 와서 "수치로 말한다." `payment-v2-stability-test`도 사후검증을 별도 글로 분리해서 설계→검증의 흐름을 보여준다. 이건 대부분의 주니어 블로그에서 안 나오는 디테일이다.

### 1.3 시리즈 내부 링크 (`{% post_url %}`)
`eda-event-driven-architecture` → `transactional-event-listener`, `payment-v2-architecture` → `payment-v2-stability-test`처럼 Jekyll의 post_url로 연결한 부분이 좋다. 다만 일관적이지 않다(아래 약점 참고).

### 1.4 도입부 인용블록 한 줄 요약
`> ...`로 시작하는 한 줄 요약이 점점 정착되고 있다 (oracle-index, distributed-counter, kotlin-pattern#1 등). 검색 유입자에게 첫 인상을 결정하는 요소이므로 모든 글로 확장해야 한다.

### 1.5 "더 파볼 키워드" 섹션
`oracle-index-structure`의 `## 9. 더 파볼 키워드`, `distributed-issue-limit-counter`의 `## 부록: 키워드` — 이게 학습 자료로서의 다음 학습 동선을 만들어준다. 다른 글에도 일반화하면 좋다.

### 2.1 알고리즘 글의 품질 편차

**문제 진단**

| 항목 | 9461 (파도반) | 13144 (투포인터) |

- 이미 쓴 알고리즘 글 16개 + 2026년 11개를 위 템플릿으로 점진 리팩토링 (1주에 2개씩)
- 시간복잡도/유형/난이도를 한 줄에 박아두면 카테고리 페이지에서 한눈에 분류된다
- "같은 유형 다른 문제" 섹션이 SEO와 학습자료 가치를 동시에 끌어올린다

### 2.2 CS/네트워킹 글의 "책 정리" 톤

**문제 진단**

`2026-02-19-application-layer.md`는 컴퓨터네트워킹 책의 2.1~2.6 챕터를 거의 그대로 정리한 노트다. 챕터 헤딩(2.1, 2.2, 2.3...)이 책의 구조를 그대로 따라가고, 본인의 관점/실습/실무 연결이 빠져 있다. **개인 노트로는 매우 좋지만, 기술 블로그로 공개하면 검색에서 책 요약본들과 경쟁해야 한다.** 이기기 어렵다.

**개선안: 책 노트를 블로그 글로 승격하는 3가지 방법**

1. **챕터 단위가 아니라 "내가 헷갈렸던 1가지" 단위로 글을 쪼갠다.**
   - 한 챕터 = 한 글이 아니라 → "HTTP/2 멀티플렉싱과 HTTP/1.1 병렬 TCP의 차이를 다시 정리한 이유" 같은 식
   - 글 수는 늘지만 각 글이 더 검색되고 더 유니크해진다
2. **실습/측정 한 컷씩 끼워넣는다.**
   - DNS 글이면 → `dig +trace google.com` 결과 캡처
   - HTTP/2 글이면 → curl `--http2`와 `--http1.1` 비교
   - 캐싱 글이면 → 브라우저 DevTools Network 탭 캡처
3. **실무 사례 / 백엔드 관점 연결.**
   - "DNS 부하분산을 클라우드 LB 앞단에서 활용하는 방식"
   - "쿠키 → 토큰 인증의 이유와 OAuth 흐름 (이미 본인이 InviteController에서 쓴 패턴과 연결)"
   - 이미 쓴 본인의 Spring 글들과 cross-link 가능

이렇게 하면 "CS 노트"가 "CS와 백엔드를 잇는 글"로 격상되고, 포트폴리오 가치가 생긴다.

### 2.3 회고 글의 부족

**문제 진단**

회고는 단 1개(`karrot-interview-retrospective`)다. 그것도 매우 잘 썼다(잘한 것/못한 것/정리의 구조, 솔직한 톤). 그런데 이게 1개뿐이라는 게 문제.

**왜 회고가 포폴에 중요한가**

| 종류 | 보여주는 능력 |
|---|---|
| 프로젝트 회고 | 의사결정 능력, 실수에서 배우는 능력 |
| 면접 회고 | 자기 객관화, 피드백 수용 |
| 분기/연말 회고 | 학습 자기관리, 목표 정렬 |
| 기술 의사결정 회고 | "이 기술을 도입했더니 6개월 뒤 OO 문제가 생겼다" — 가장 시니어다움 |

**개선안**

- **`Retrospect` 카테고리를 1급 카테고리로 격상.** 현재 `[retrospective]` (소문자/오타)로 들어있고 CLAUDE.md의 6대 카테고리에 없다.
- 최소 분기별 1회 회고 (3, 6, 9, 12월)
- "이 글을 6개월 후에 다시 읽으면 뭐가 바뀌어 있을까?" 시점 글 — 미래의 자신과 대화
- payment-v2 / vault-ledger / WaitFair 프로젝트마다 **별도 회고 글 1개씩** — "다시 한다면 어떻게 할까" 형식. 면접에서 그대로 답변이 된다.

### 2.4 메타데이터 / SEO / 발견성

**문제 진단**

front matter를 보면 거의 모든 글에 `description` 필드가 없다. Chirpy는 description을 메타태그/OG/검색 미리보기에 쓰는데, 이게 없으면 글 첫 줄이 잘려서 노출된다.

**구체적 결함 목록**

| 결함 | 영향 |
|---|---|
| `description` front matter 누락 (전 포스트) | 검색/소셜 카드 품질 저하 |
| `image` front matter 누락 (썸네일 OG) | 트위터/슬랙 공유 시 미리보기 빈약 |
| 외부 이미지(`postimg.cc`)에 의존 | 호스팅 만료/링크 데드 위험. 5년 후 글이 깨질 수 있다 |
| 디렉토리명 오타: `_posts/nerworking/` | 일관성 깨짐, IDE/스크립트에서 실수 유발 |
| 파일명 `kotlin-pattern#1.md` (해시 포함) | URL slug 인코딩, 다른 정적 사이트 빌더로 마이그레이션 시 문제 |
| 카테고리 불일치: CLAUDE.md엔 6개인데 실제로는 `[CS]`, `[Blockchain]`, `[retrospective]` 등 추가 | 사이드바 카테고리 카운트 부정확 |
| 이미지 alt 텍스트 부재 | 접근성·SEO 모두 손해 |
| 태그 표기 혼재: `problem-solving` / `algorithm` / `boj` 셋 다 사용 | 태그 페이지 분산 |

**개선안: 1회성 정리 작업**

1. **`assets/img/posts/<slug>/` 구조로 이미지 로컬화.**
   - 기존 postimg 링크는 `wget`으로 일괄 다운로드 → 로컬 이동 → 본문 경로 치환 스크립트로 일괄 변환
2. **모든 post에 `description` 필드 추가.**
   - 첫 인용블록의 한 줄 요약을 그대로 description으로 복사하는 스크립트로 60% 자동화 가능
3. **카테고리 정식화: CLAUDE.md를 9개로 업데이트**
   - Spring, React, Algorithm, Kotlin, DevOps, Database, **CS, Blockchain, Retrospect**
   - 추가로 **`Project` 카테고리 신설** (아래 2.5 참고)
4. **디렉토리/파일명 정리**
   - `_posts/nerworking/` → `_posts/cs/`
   - `kotlin-pattern#1.md` → `kotlin-pattern-1.md` (또는 `-part1`)
   - `algo-11055.md` → `boj-11055-가장-큰-증가-부분수열.md` 처럼 검색 가능한 슬러그
5. **태그 표준 문서 만들기.** CLAUDE.md에 태그 사전을 추가하고 lint 스크립트로 검증.

### 2.5 포트폴리오 가시성 — "이미 잘 쓴 글이 잘 안 보인다"

**문제 진단**

`payment-v2-architecture`, `concurrency-comparison-test`, `distributed-issue-limit-counter`, `vault-ledger-learning-guide`는 그 자체로 강력한 포폴 자산이다. 그런데 사이드바에서 보면 그냥 Spring/Blockchain 카테고리에 묻혀 있다.

**개선안: "Project" / "기술 의사결정" 톱레벨 카테고리 신설**

Chirpy의 `_tabs/`에 커스텀 페이지를 추가해서:

```
/projects
  ├ WaitFair (티켓팅) — 7개 글 cross-link, GitHub repo, 아키텍처 다이어그램, 내 역할
  ├ vault-ledger (블록체인 인덱서) — 1개 글, repo, 학습 가이드
  └ ... (앞으로 추가)
```

각 프로젝트 페이지는 다음 구조:

```markdown
# WaitFair — 동시성에 강한 콘서트 티켓팅 시스템

- 결제 V2 아키텍처 설계 및 구현
- 동시성 제어 전략 비교·선정 (CAS + Optimistic Lock 채택)
- 장애 대응 (Circuit Breaker, 멱등성, 사후 안정성 테스트)

## 핵심 기술 의사결정
| 결정 | 대안 | 채택 이유 | 관련 글 |
|---|---|---|---|
| CAS + Optimistic Lock | Redisson, Opt-only | tail latency 안정 | [→ 비교실험](.../concurrency-comparison-test) |
| PG 호출과 DB 트랜잭션 분리 | @Transactional로 감싸기 | 외부 장애 격리 | [→ 아키텍처](.../payment-v2-architecture) |
| ...

## 핵심 글
- [Payment V2 아키텍처](...) — pin: true
- [동시성 비교실험](...) — pin: true
- [결제 무결성 사후검증](...)
- [Redisson 분산 락](...)

## 결과/지표
- p99 응답시간: 173ms (좌석선택 VU=100)
- 무결성 검증 통과 (중복결제 0건, 멱등성 3단계 방어)

## 링크
- GitHub: <repo URL>
- 아키텍처 다이어그램 (PNG)
- 데모 영상 (선택)
```

면접관에게 "포트폴리오 한 페이지 보세요"라고 줄 수 있는 형태. 이게 없으면 면접관이 글 30개를 뒤지면서 의사결정을 찾아야 한다.

**`pin: true` 활용도 확대.** 현재 `payment-v2-architecture`와 `concurrency-comparison-test` 두 개가 pin이다. 그 외에 `oracle-index-structure`, `distributed-issue-limit-counter`도 pin 가치가 충분히 높다. 단, pin 글이 늘면 hierarchy가 무너지니까 **프로젝트 페이지로 1차 진입 → 글 상세로** 가는 동선이 더 좋다.

### 2.6 글쓰기 디테일 — 모든 글에 적용할 5가지

1. **TL;DR / 글 끝의 "한 줄 교훈"을 의무화**
   - oracle-index의 "한 장 요약" 표, distributed-counter의 "한 줄 요약/일반화" — 이 형식을 표준화.
2. **코드 블록 앞에 "왜 이렇게 짰는가" 한 줄**
   - 좋은 예: kotlin-pattern#1의 `verifier.verify(...)` 호출 위에 "동일 타입 파라미터의 버그 방어"
   - 코드만 있으면 학습자료로 무게가 안 실린다
3. **재현 가능성 명세 (성능 실험 글 한정)**
   - JDK 버전, CPU, RAM, DB 인스턴스 타입, k6 스크립트 링크
   - 이게 없으면 수치의 신뢰도가 떨어진다
4. **시리즈 navi 박스 표준화**
   - 글 상단: `> 이 글은 Kotlin 패턴 시리즈 [1/4] 입니다. → [#2](...) → [#3](...) → [#4](...)`
   - Liquid include로 한 번만 정의하면 시리즈 글에 재사용 가능
5. **본인 의견 글 비중 늘리기**
   - 지금은 "사실/설계/분석"이 95%, "내 의견/주장"이 5%다
   - "왜 나는 Spring보다 Kotlin Spring을 추천하는가" / "Redisson을 안 쓰기로 결정한 이유" / "왜 OLTP에서 Bitmap을 안 쓰는가" 같은 1인칭 주장 글
   - 의견 글은 댓글·반응을 만들고 시그니처를 만든다

### 2.7 학습자료로서의 가치 증폭

이미 oracle/payment 시리즈는 강의 자료에 가깝다. 이걸 좀 더 학습자료답게 만드는 디테일:

| 항목 | 적용 |
|---|---|
| **체크리스트 박스** | "이 글을 다 읽고 나면 다음을 설명할 수 있어야 한다: ☐ ROWID의 4개 필드 ☐ PCTFREE를 낮게/높게 잡는 기준 ☐ ..." |
| **퀴즈/생각해보기 박스** | 본문 중간에 "잠깐, 왜 leaf끼리 linked list가 있을까?" (스포일러 토글) |
| **오개념 박스** | "흔히 ㅁㅁ라고 오해하는데, 사실은..." 일종의 안티패턴 박스 |
| **참고 자료 등급화** | 공식 docs > 책 > 신뢰할만한 블로그 순으로 분리. 지금은 다 같이 섞여 있음 |
| **Mermaid 다이어그램** | ASCII art도 좋지만 `mermaid` 코드블록으로 그리면 PNG로 export·재사용도 쉽다. Chirpy가 mermaid 지원함 |
