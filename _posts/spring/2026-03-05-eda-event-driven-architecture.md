> MSA를 도입하면 확장성과 유연성을 얻지만, 동시에 새로운 복잡성을 갖게 된다. EDA는 그 복잡성을 줄이는 아키텍처 패턴이다.

이 글은 스터디에서 EDA를 발표하며 정리한 내용을 블로그 형태로 옮긴 것이다. 개념 이해를 먼저 다루고, 실제 프로젝트(WaitFair)에서 Spring Event Bus를 활용한 애플리케이션 레벨 EDA 적용기는 [별도 포스팅]({% post_url 2026-03-05-transactional-event-listener %})에서 이어진다.

---

## 왜 EDA를 공부하게 됐나

WaitFair 프로젝트에서 좌석 상태 변경 시 WebSocket 알림을 발행하는 구조를 설계하다가, 자연스럽게 "이벤트 기반으로 결합도를 낮추는 방식"을 찾게 됐다. Spring의 `@EventListener`를 적용하면서 "이게 EDA의 일부구나"라는 감이 왔고, 개념을 제대로 정리하고 싶어서 스터디 발표 주제로 선택했다.

---

## 먼저: MSA가 가져오는 복잡성

EDA를 이해하려면 먼저 MSA(Micro Service Architecture)의 문제점을 알아야 한다.

MSA는 하나의 애플리케이션을 **독립적으로 배포 가능한 서비스 단위로 분할**하여 각 서비스가 독립적으로 확장되고 변경될 수 있게 하는 구조다. 확장성과 유연성 측면에서 강력하지만, 서비스가 분산되면서 아래와 같은 문제가 따라온다.

| 문제 | 설명 |
|---|---|
| **서비스 간 통신 복잡성** | 서비스끼리 직접 통신하면서 네트워크 지연, 통신 오류가 발생하고 의존 관계가 복잡해진다 |
| **데이터 일관성** | 각 서비스가 독립적인 DB를 가지므로 전체 시스템의 데이터 동기화가 어렵다 |
| **분산 트랜잭션** | 여러 서비스에 걸친 트랜잭션을 원자적으로 처리하기 어렵다 |
| **서비스 디스커버리** | 서비스가 많아지면 각 서비스의 위치를 동적으로 파악해야 한다 |
| **보안/권한 관리** | 분산 환경에서의 인증·인가 체계가 복잡해진다 |

이 중에서 **통신 복잡성과 데이터 일관성** 문제를 EDA가 효과적으로 다룬다.

## EDA(Event-Driven Architecture)란

> 분리된 서비스 또는 시스템들 간에 **이벤트를 게시(Publish), 소비(Subscribe), 라우팅**하는 방식으로 정보를 공유하는 아키텍처.

핵심은 **이벤트 생성자와 소비자가 서로를 직접 알지 못한다**는 것이다. 생성자는 이벤트를 발행하고 끝이고, 어떤 소비자가 받아서 무엇을 하는지 관심이 없다. 이 구조가 느슨한 결합(Loose Coupling)을 만든다.

### 이벤트(Event)란

이벤트는 **시스템 또는 서비스의 상태 변화를 나타내는 신호**다. 몇 가지 중요한 특징이 있다.

- **불변성**: 이미 발생한 일의 기록이므로 수정하거나 삭제할 수 없다
- **독립성**: 이벤트 발행 시점에 소비자 로직에 관계없이 발생한다
- **재처리 가능성**: Kafka 같은 이벤트 브로커는 이벤트를 영구 저장하여 과거 시점부터 Replay가 가능하다

예시: 사이트 방문, 주문 생성, 결제 완료, 좌석 상태 변경 등 비즈니스 내외부에서 발생하는 모든 사건.

이벤트 포맷의 표준도 있다. Google Cloud의 Eventarc는 CloudEvents 스펙을 따르는 JSON 형식을 사용한다.

```json
{
  "data": { "seat_id": 42, "status": "RESERVED" },
  "datacontenttype": "application/json",
  "id": "MESSAGE_ID",
  "source": "//ticketing.service/seats",
  "specversion": "1.0",
  "type": "seat.status.changed",
  "time": "2026-01-07T09:00:00Z"
}
```

## EDA의 구성 요소

### 이벤트 버스(Event Bus)

이벤트 기반 아키텍처의 핵심 구성 요소로 **시스템 간 이벤트 통신을 중개**하는 역할을 한다.

비슷한 용어들이 혼용되는데 구분하면 이렇다.
- **이벤트 버스**: 단일 애플리케이션 또는 소규모 시스템 내부의 이벤트 전달 메커니즘 (예: Spring `ApplicationEventPublisher`)
- **이벤트 브로커/라우터**: 여러 애플리케이션, 서비스, 네트워크를 넘나드는 더 넓은 범위의 이벤트 관리 (예: Kafka, RabbitMQ, AWS EventBridge)

### 이벤트 채널(Event Channel)

이벤트 생성자와 소비자 사이의 **논리적 통신 경로**다. 생성자는 특정 채널에 이벤트를 발행하고, 그 채널을 구독(Subscribe)한 소비자들이 이벤트를 수신한다.

### EDA의 전체 흐름

[![image-(1).png](https://i.postimg.cc/76SnzHBb/image-(1).png)](https://postimg.cc/qhMKVTHd)

예를 들어 "주문 생성" 이벤트가 발행되면 이벤트 버스가 이를 수신하는 DB, 결제 서비스, 알림 서비스에 각각 라우팅한다. 각 소비자는 서로를 모르고 독립적으로 처리한다.

## EDA가 MSA 문제를 어떻게 해결하는가

### 1. 서비스 간 통신 복잡성 완화

서비스들이 직접 서로를 호출하는 대신 **이벤트 라우터를 통해 간접 통신**한다. 서비스가 늘어나도 새로운 소비자를 구독 목록에 추가하기만 하면 된다.

단, 이 구조의 **트레이드오프**가 있다. 서비스 간 직접 의존성은 줄어들지만, 전체 흐름을 추적하기가 어려워진다.

```
사용자 → 주문 버튼 클릭
   └→ 주문 생성 이벤트
         └→ 결제 진행 이벤트
               └→ 재고 차감 이벤트  ← 여기서 에러 발생
                     └→ 배송 준비 이벤트
```

위 흐름에서 재고 차감에서 에러가 났을 때, 각 서비스에 흩어진 로그를 하나로 이어붙여야 디버깅이 가능하다. 이를 위해 **분산 추적 시스템(Distributed Tracing)** 이 별도로 필요하다.

- **단일 애플리케이션 내 추적**: MDC(Mapped Diagnostic Context) — 하나의 서버 안에서 특정 요청이 들어왔을 때 해당 스레드의 모든 로그에 동일한 `requestId`를 기록
- **서비스 간 추적**: Jaeger, Zipkin, Grafana Tempo — `A 서버 → 이벤트 라우터 → B 서버`로 이어지는 전체 흐름에 `Trace ID`를 부여

> WaitFair에서는 단일 어플리케이션이었기 때문에 MDC 기반 requestId 추적을 적용했다. MDC에 requestId, userId, eventId, seatId, orderId를 담아 장애 원인 분석 시 로그를 요청 단위로 묶어서 볼 수 있게 했다.

### 2. 데이터 일관성 확보 (최종 일관성)

MSA에서 가장 어려운 문제인 분리된 서비스 간 데이터 동기화를 이벤트 라우터를 통해 **최종 일관성(Eventual Consistency)** 방식으로 지원한다.

단, EDA만으로 완벽한 데이터 일관성이 보장되지는 않는다. 이벤트 발행 중 유실이나 처리 실패가 생길 수 있기 때문에 **Saga 패턴**, **보상 트랜잭션(Compensating Transaction)** 등이 함께 사용된다.

### 3. 트랜잭션 관리 단순화

이벤트 기반으로 동작하면 각 서비스가 자신의 영역만 책임진다. 생성자는 이벤트를 발행하고, 소비자는 자신의 처리만 한다. 전체를 아우르는 분산 트랜잭션 없이도 각자의 로컬 트랜잭션으로 처리가 가능해진다.

### EDA의 추가 이점

**느슨한 결합**: 각 서비스는 기술 스택에 상관없이 JSON 이벤트 형식만 맞추면 된다. Java 서비스가 발행한 이벤트를 Python 서비스가 소비할 수 있다.

**모니터링**: 서비스가 서로를 Polling하며 상태를 확인할 필요가 없다. 에러 이벤트를 라우터에 던지면 관리자는 라우터에 모이는 이벤트만 관찰하면 된다.

**팬아웃(Fan-out) 및 병렬 처리**: 하나의 이벤트를 여러 소비자에게 동시에 전달할 수 있다. 결제 완료 이벤트 하나가 알림 서버, 통계 서버, 배송 서버를 동시에 독립적으로 트리거한다.

## EDA의 세 가지 모델

### 1. Pub/Sub 모델 — 1:N 브로드캐스팅

생성자가 특정 토픽(Topic)에 이벤트를 발행하면, 해당 토픽을 구독 중인 **모든 소비자가 동시에** 수신한다.

- **언제 쓰나**: 결제 완료 시 알림 서버, 통계 서버, 배송 서버 모두에 동일한 이벤트를 동시에 전달해야 할 때
- **대표 기술**: Google Cloud Pub/Sub, AWS SNS, Redis Pub/Sub

### 2. 메시지 큐 모델 — 1:1 작업 분산

생성자가 큐(Queue)에 메시지를 넣으면, 여러 소비자 중 **단 하나**만 해당 메시지를 가져가서 처리한다. 처리 완료 후 메시지는 큐에서 삭제된다.

- **언제 쓰나**: 이미지 리사이징, 이메일 발송처럼 트래픽이 몰릴 때 작업을 여러 워커에 분산해서 순차 처리할 때
- **대표 기술**: RabbitMQ, AWS SQS

### 3. 이벤트 스트리밍 모델 — 연속적 로그 기록

이벤트를 소비 즉시 삭제하지 않고 **디스크에 로그 형태로 순차 보관**한다. 소비자는 자신의 처리 속도에 맞춰 이벤트를 읽어가고, 장애 발생 시 과거 시점부터 이벤트를 다시 읽어 재처리(Replay)할 수 있다.

- **언제 쓰나**: 대용량 실시간 스트림 처리, 이벤트 감사 이력이 필요하거나, 장애 후 복구가 중요한 시스템
- **대표 기술**: Apache Kafka, AWS Kinesis, Redis Streams

## 세 모델 비교

| | Pub/Sub | Message Queue | Event Streaming |
|---|---|---|---|
| **메시지 수신** | 구독자 전체 (1:N) | 단일 소비자 (1:1) | 소비자별 독립 오프셋 |
| **메시지 보존** | 전달 후 소멸 | 처리 후 삭제 | 기간 동안 디스크 유지 |
| **Replay** | 불가 | 불가 | 가능 |
| **주 용도** | 팬아웃, 알림 | 작업 분산, 부하 조절 | 감사 이력, 대용량 스트림 |

## 정리

EDA는 MSA의 강한 결합 문제를 **이벤트를 매개로 느슨하게 연결**하는 방식으로 풀어낸다. 완벽한 해결책은 아니고, 추적 복잡도와 최종 일관성이라는 트레이드오프가 따른다.

개념 레벨에서 중요한 것:
- 이벤트는 불변이고, 발생자와 소비자는 서로를 모른다
- EDA 구조 자체는 데이터 일관성을 보장하지 않는다 → Saga, 보상 트랜잭션 필요
- 분산 추적 없이는 디버깅이 어려워진다 → MDC(단일 앱), Jaeger/Zipkin(분산)

다음 포스팅에서는 이 개념을 단일 Spring Boot 애플리케이션 내부에 적용한 사례, 즉 `@TransactionalEventListener`를 이용해 좌석 도메인과 WebSocket 발행을 분리한 경험을 다룬다.
