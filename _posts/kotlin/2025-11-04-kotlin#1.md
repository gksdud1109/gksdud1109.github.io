## 왜 Lombok은 코틀린에서 지원 안 함?

👉 결론: 코틀린은 언어 자체에서 이미 Lombok이 하던 걸 “내장 문법”으로 해결하기 때문.

자바는 보일러플레이트(반복 코드)가 많아서 Lombok이 필요했음.

예를 들어:

```java
@Getter
@AllArgsConstructor
public class User {
    private final String name;
    private final int age;
}
```

이걸 코틀린으로 바꾸면

```kotlin
data class User(val name: String, val age: Int)
```

### 이미 자동으로
-	getter / toString() / equals() / hashCode() / copy()까지 제공함
-	생성자도 자동
-	val/var로 불변성까지 명확하게 구분

그래서 Lombok을 쓸 필요도, 써서 얻을 이득도 없고, 오히려 @Getter, @RequiredArgsConstructor 같은 Lombok 애노테이션이 IDE 분석기를 방해할 수도 있음.

📌 정리
코틀린은 Lombok이 언어에 녹아있다.
즉, “Lombok을 지원 안 하는 게 아니라, 필요 없어서 사용하지 않는다.”

# Kotlin에서 기본 생성자

```kotlin

class CustomOAuth2LoginSuccessHandler(
    private val memberService: MemberService,
    private val rq: Rq
) : AuthenticationSuccessHandler {
    // ...
}
```

위에 있는게 **주 생성자(primary constructor)** 임.
코틀린 클래스는 클래스 이름 옆에 오는 괄호 속의 인자들이 생성자 파라미터가 됌.
이 안에 val이나 var을 붙이면, 자동으로 필드가 되고 DI 대상이 돼.

자바로 치면 아래랑 같은 의미👇

```java
@Component
public class CustomOAuth2LoginSuccessHandler implements AuthenticationSuccessHandler {
    private final MemberService memberService;
    private final Rq rq;

    public CustomOAuth2LoginSuccessHandler(MemberService memberService, Rq rq) {
        this.memberService = memberService;
        this.rq = rq;
    }
}
```
즉, 코틀린은 기본 생성자 안에서 의존성 주입까지 한 줄로 끝나는 구조!

# !!는 무슨 뜻이야?

> 👉 “나는 이 값이 null이 아닐 거라고 확신하니까 그냥 꺼내줘.” </br>
하지만, 그게 만약 null이면 NPE(NullPointerException) 발생.

예시: 

```kotlin
val name: String? = null
println(name!!.length)  // ❌ NullPointerException
```

-	!!는 nullable 타입(String?)을 non-null로 강제 변환해버림.
-	실제로 null이면 즉시 런타임 오류.
-	그래서 “나는 절대 null 아닐 거야”라는 확신이 있을 때만 써야 하는데, 대부분의 경우 안전한 null 처리 (?., ?:)로 대체 가능.

### 📌 원칙
- ?. → null-safe 접근
- ?: → null일 때 기본값
- !! → 위험, 최대한 쓰지 말기

<br>

# 타입 뒤의 ?는 무슨 뜻이야?

> 👉 그 타입이 null이 될 수 있다는 뜻.

```kotlin
val name: String? = null   // 가능
val name2: String = null   // ❌ 불가능
```

즉, String?은 nullable,
String은 non-null.

자바에서는 기본적으로 모든 참조 타입이 nullable이라서 컴파일러가 확인해주지 않지만,
코틀린은 타입 시스템 차원에서 null 안전을 보장해줌.

# isNullOrBlank()와 ? 연산자의 관계

베이스 코드
```kotlin
private fun extractRedirectUrl(state: String?): String? {
    if (state.isNullOrBlank()) return null
    val decoded = runCatching {
        String(Base64.getUrlDecoder().decode(state), Charsets.UTF_8)
    }.getOrNull() ?: return null

    // "#" 기준으로 안전하게 분리
    val parts = decoded.split("#")
    return parts.getOrNull(1)
}
```

여기서 이부분👇
```kotlin
if (state.isNullOrBlank()) return null
```

여기서
-	state는 String? (즉, null일 수도 있는 문자열)
-	isNullOrBlank()는 확장 함수로,
내부에서 null인지 자동으로 체크해주는 함수.

정의는 이렇게 되어 있음 (표준 라이브러리):

```kotlin
fun CharSequence?.isNullOrBlank(): Boolean {
    return this == null || this.isBlank()
}
```
즉,
-	state가 null이면 true
-	state가 "   " (공백 문자열)이라도 true
-	나머지는 false

따라서 if (state.isNullOrBlank()) return null은
“null이거나 공백이면 함수 종료”라는 의미임.

즉, 여기선 ?.이 아니라 isNullOrBlank() 자체가 null-safe 함수라 ? 필요 없음.


# runCatching은 뭐야?

베이스 코드
```kotlin
private fun extractRedirectUrl(state: String?): String? {
    if (state.isNullOrBlank()) return null
    val decoded = runCatching {
        String(Base64.getUrlDecoder().decode(state), Charsets.UTF_8)
    }.getOrNull() ?: return null

    // "#" 기준으로 안전하게 분리
    val parts = decoded.split("#")
    return parts.getOrNull(1)
}
```
엘비스(?:)는 어떻게 작동해?

## ① runCatching

표준 라이브러리 함수임 (kotlin.Result API).
어떤 블록 안의 예외를 try-catch 없이 깔끔하게 감싸서 결과로 돌려줌.

```kotlin
val result = runCatching {
    riskyOperation() // 예외가 발생할 수도 있는 코드
}
```

그 결과로 Result<T> 객체를 반환해서 이렇게 쓸 수 있음 👇
```kotlin
val value = runCatching {
    riskyOperation()
}.getOrNull() // 예외 나면 null, 성공하면 결과 반환
```

즉, try-catch의 간결한 대체 문법!

## ② 엘비스 연산자 (?:)

?:는 “null이면 오른쪽 값을 써라” 라는 뜻.

```kotlin
val name: String? = null
val displayName = name ?: "이름없음"
println(displayName) // 👉 "이름없음"
```

즉,
-	?: 왼쪽이 null이면 → 오른쪽 값 사용
-	왼쪽이 non-null이면 → 그대로 사용

## 결과적으로 베이스 코드에서의 runCatching 부분 동작은

```kotlin
val decoded = runCatching {
    String(Base64.getUrlDecoder().decode(state), Charsets.UTF_8)
}.getOrNull() ?: return null
```

> “Base64 decode 하다가 에러 나면 null을 반환하고, 그럼 함수 종료시켜라.”


# 🎯 기본 문법 요약 표

| 문법 | 의미 | 자바에서의 대응 |
|:----|:----|:----|
|? | null 허용 타입 | Optional or @Nullable|
|!! | 강제 non-null (null이면 NPE) | 그냥 NPE|
|?. | null-safe 접근 | if (obj != null) { obj.method() }|
|?: | null일 때 기본값 지정 | obj != null ? obj : default |
|isNullOrBlank() | null + 공백 동시 체크 | obj == null |
|runCatching {} | 예외 감싸기 | try-catch 블록 |
