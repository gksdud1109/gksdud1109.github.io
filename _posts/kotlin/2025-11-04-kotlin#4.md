```kotlin
class Ut {
    object jwt {

        @JvmStatic
        fun toString(secret: String, expireSeconds: Long, body: Map<String, Any>): String {
            val issuedAt = Date()
            val expiration = Date(issuedAt.time + 1000L * expireSeconds)
            val secretKey: Key = Keys.hmacShaKeyFor(secret.toByteArray(StandardCharsets.UTF_8))

            return Jwts.builder()
                .claims(body)
                .issuedAt(issuedAt)
                .expiration(expiration)
                .signWith(secretKey)
                .compact()
        }

        fun isValid(jwt: String?, secretPattern: String): Boolean{
            if(jwt.isNullOrBlank()) return false

            val secretKey = Keys.hmacShaKeyFor(secretPattern.toByteArray(StandardCharsets.UTF_8))
            return runCatching {
                Jwts.parser()
                    .verifyWith(secretKey)
                    .build()
                    .parseSignedClaims(jwt)
            }.isSuccess
        }

        fun payloadOrNull(jwt: String?, secretPattern: String): Map<String, Any>?{
            if(jwt.isNullOrBlank() || !isValid(jwt,secretPattern)) return null

            val secretKey = Keys.hmacShaKeyFor(secretPattern.toByteArray(StandardCharsets.UTF_8))
            return runCatching {
                val parsed = Jwts.parser()
                    .verifyWith(secretKey)
                    .build()
                    .parseSignedClaims(jwt)

                parsed.payload as Map<String, Any>
            }.getOrNull()
        }
    }
}

```

# 자바에서의 정적 유틸 구조란?

자바에서는 공용(static) 메서드만 있는 유틸리티 클래스를 자주 만든다.

```java
public class MathUtil {
    public static int add(int a, int b) {
        return a + b;
    }
}
```

-	MathUtil.add(3, 4) 이렇게 클래스 이름으로 바로 호출 가능
-	객체 생성(new MathUtil())이 필요 없음
-	이런 걸 “정적 유틸리티 클래스”라고 부름

# 그런데 Kotlin에는 static이 없음

Kotlin은 static 키워드를 없애고 대신 object 키워드로 싱글턴(Singleton) 객체를 정의함.
즉, object는 클래스 선언 + 인스턴스 1개 생성이 동시에 일어나는 개념임.

## Kotlin에서의 정적 유틸 구조

자바의 static 유틸 클래스 대신 Kotlin에서는 이런 식으로 정의함 👇

```kotlin
object MathUtil {
    fun add(a: Int, b: Int): Int = a + b
}
```

➡️ 이렇게 하면,
-	객체를 만들지 않아도 (new 불필요)
-	언제든 바로 MathUtil.add(3, 4) 호출 가능
-	내부적으로는 MathUtil이라는 전역 싱글턴 객체로 관리됨

## 예시코드 구조에서는

```kotlin
class Ut {
    object jwt {
        fun toString(...) { ... }
        fun isValid(...) { ... }
        fun payloadOrNull(...) { ... }
    }
}
```
이건 자바로 치면 아래와 같은 구조인거임👇

```java
public class Ut {
    public static class jwt {
        public static String toString(...) { ... }
        public static boolean isValid(...) { ... }
        public static Map<String, Object> payloadOrNull(...) { ... }
    }
}
```
-	Ut는 최상위 “유틸 클래스”
-	jwt는 그 안의 “정적 유틸 객체”
-	호출할 때는 Ut.jwt.toString()처럼 정적 접근이 가능


## 정리 — Kotlin에서 object의 역할

| 구분 | Java | Kotlin |
|:---|:---|:---|
|static 메서드 | public static void foo()| object Foo { fun bar() }|
|호출 방식 | Foo.bar() | Foo.bar() (동일) |
|메모리 생성 | 클래스 로딩 시 생성되지 않음 (정적) | 런타임 시 최초 접근 시점에 1회 생성 (싱글턴) |
|객체 생성 필요 여부 | ❌ 없음 | ❌ 없음 |

## @JvmStatic을 지우면 toString을 호출할 수 없음

왜냐면

```
Ut.jwt.toString(secretPattern, expireSeconds, Map.of(...));
```

이 코드가 "정적 메서드 호출" 처럼 해석되지만, kotlin에서는 object 내부의 함수는 실제로는 정적이 아님

## Kotlin의 object 내부 함수는 실제로 이렇게 컴파일됌

```java
class Ut {
    object jwt {
        fun toString(...) = ...
    }
}
```
JVM 바이트코드로 보면 대략 이런 형태로 변환됌👇
```
public final class Ut {
    public static final class jwt {
        public static final jwt INSTANCE = new jwt();

        public String toString(String secret, long expireSeconds, Map<String, Object> body) {
            ...
        }
    }
}
```

즉,
-	jwt는 실제로 클래스(Ut$jwt) + 전역 인스턴스(INSTANCE)로 만들어지고
-	그 안의 toString()은 인스턴스 메서드로 존재함.

그래서 자바에서 호출하려면 이렇게 해야함 👇
```java
Ut.jwt.INSTANCE.toString(...);
```

그래서 @JvmStatic이 없으면 안된다.

## 위와 같이 되는 이유: 

### Object는 Kotlin의 런타임 싱글턴


```kotlin
object JwtUtil {
    fun encode() = println("encode")
}
```

```java
public final class JwtUtil {
    public static final JwtUtil INSTANCE = new JwtUtil(); // ← 여기서 싱글턴 보장됨

    private JwtUtil() {} // 생성자는 private

    public final void encode() {
        System.out.println("encode");
    }
}

```

💡 즉, object는
-	컴파일 시 **클래스 1개(JwtUtil)** 와
-	내부에 INSTANCE 정적 필드 1개가 만들어짐.
-	이 필드에 단 하나의 인스턴스가 담김.

그래서 Kotlin에서 JwtUtil.encode()라고 쓰면 <br>
→ 컴파일러가 내부적으로 JwtUtil.INSTANCE.encode()로 바꿔줌.<br>
(Kotlin 컴파일러가 자동으로 “static처럼” sugar syntax로 바꿔주는 거임.)

### 자바에서 부르면 static처럼 안 되는 이유

Kotlin 내부에서는 sugar syntax (JwtUtil.encode())를 제공하지만
Java는 Kotlin의 문법을 모름.

그래서 Java에서 보면 **“encode는 인스턴스 메서드”** 인 거임.
즉, 이렇게 해야 접근 가능 👇

```java 
JwtUtil.INSTANCE.encode();
```

### 그럼 @JvmStatic은 왜 필요한가?

@JvmStatic은 “이 함수를 진짜 static 메서드로도 동시에 생성하라”는 의미임.

예를 들어 👇

```kotlin
object JwtUtil {
    @JvmStatic
    fun encode() = println("encode")
}
```
그럼 컴파일된 자바 바이트코드에서는 이렇게 됌👇
```
public final class JwtUtil {
    public static final JwtUtil INSTANCE = new JwtUtil();

    private JwtUtil() {}

    // 원래의 인스턴스 메서드
    public final void encode() { ... }

    // 추가로 생성된 진짜 static 메서드
    public static void encode() { 
        INSTANCE.encode(); 
    }
}
```

# companion object

companion object는 **클래스 내부의 “정적 영역”** 을 흉내 내는 구조임.

예를 들어 👇
```kotlin
class JwtUtil {
    companion object {
        fun encode() = println("encode")
    }
}
```

바이트코드로는 이렇게 나옴👇
```java
public final class JwtUtil {
    public static final class Companion {
        public void encode() { ... }
    }

    public static final Companion Companion = new Companion();  // ← 여기에 인스턴스 저장
}
```

즉,
-	JwtUtil.Companion.encode() 이렇게 접근해야 함
-	Kotlin에서는 JwtUtil.encode()로 sugar 처리됨 (마치 static처럼)
-	Java에서는 JwtUtil.Companion.encode()로만 접근 가능 ❌

자바에서 그대로 JwtUtil.encode()로 부르고 싶다면
→ 여기도 @JvmStatic을 붙여야 함.

```kotlin
class JwtUtil {
    companion object {
        @JvmStatic
        fun encode() = println("encode")
    }
}
```
