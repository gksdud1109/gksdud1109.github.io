## 자바에서 바꾼 코틀린 코드

```kotlin
@Component
class CustomOAuth2AuthorizationRequestResolver(
    private val clientRegistrationRepository: ClientRegistrationRepository
) : OAuth2AuthorizationRequestResolver {

    private val defaultResolver = DefaultOAuth2AuthorizationRequestResolver(
        clientRegistrationRepository,
        OAuth2AuthorizationRequestRedirectFilter.DEFAULT_AUTHORIZATION_REQUEST_BASE_URI
    )

    override fun resolve(request: HttpServletRequest): OAuth2AuthorizationRequest? =
        defaultResolver.resolve(request)?.let{customizeState(it, request)}

    override fun resolve(request: HttpServletRequest, clientRegistrationId: String?): OAuth2AuthorizationRequest? =
        defaultResolver.resolve(request, clientRegistrationId)?.let{customizeState(it, request)}

    private fun customizeState(
        authorizationRequest: OAuth2AuthorizationRequest,
        req: HttpServletRequest
    ): OAuth2AuthorizationRequest {
        val redirectUrl = req.getParameter("redirectUrl") ?: "/"
        val originState = authorizationRequest.state
        val newState = "$originState#$redirectUrl"
        val encodedState = Base64.getUrlEncoder().encodeToString(newState.toByteArray(StandardCharsets.UTF_8))

        return OAuth2AuthorizationRequest.from(authorizationRequest)
            .state(encodedState)
            .build()
    }
}
```

# 여기서 OAuth2AuthorizationRequest 이게 왜 불변객체야?

## “불변 객체”

> 불변 객체(Immutable Object) : 한 번 생성되면 내부 상태(필드 값)가 절대 변경되지 않는 객체

즉, setter가 없고, 필드가 final이며, 객체의 메서드가 내부 값을 수정하지 않음.
👉 새로운 값을 가지려면 새 객체를 만들어야 함.

예시: 
```kotlin
data class User(val name: String, val age: Int)

// 기존 객체 변경이 아니라 복제(copy)
val user2 = user.copy(age = 30)
```

User는 불변 객체다.
user.age = 30처럼 바꾸는 건 불가능.

# OAuth2AuthorizationRequest 내부를 보면

```kotlin
public final class OAuth2AuthorizationRequest implements Serializable {
    private final String authorizationUri;
    private final String clientId;
    private final String redirectUri;
    private final Set<String> scopes;
    private final String state;
    private final Map<String, Object> additionalParameters;
    ...
}
```

핵심 키워드 👇
-	public final class → 상속 불가
-	모든 필드가 private final
-	setter 없음
-	생성자는 private
-	값 변경은 .from(existing).state(newValue).build() 로만 가능

즉, Spring Security가 명시적으로 불변 객체로 설계한 것임.

## 왜 이렇게 설계했을까?

OAuth2 프로토콜은 보안이 핵심
특히 OAuth2AuthorizationRequest는 사용자의 인가 요청 정보를 담고 있어서 중간에 값이 변하면 보안상 문제가 생김

예를 들어 redirectUri, clientId, state 등이 변조되면
→ OAuth 인증 흐름이 깨지고, 피싱 공격이 가능해져.

그래서 이 객체는
- 생성 시점에만 값이 정해지고,
- 이후 절대 바꿀 수 없도록 final + private constructor로 막아둔 거야.

# 그럼 .from(...).state(...).build()는 뭐야?

이건 복사 기반 빌더 패턴(copy builder)
즉, 기존 불변 객체의 내용을 기반으로 새로운 불변 객체를 생성하는 방법.

```kotlin
val newRequest = OAuth2AuthorizationRequest.from(oldRequest)
    .state("encodedState")
    .build()
```

자바 코드를 보면 from() 메서드는 내부적으로 이렇게 생김👇
```java
public static Builder from(OAuth2AuthorizationRequest authorizationRequest) {
    Assert.notNull(authorizationRequest, "authorizationRequest cannot be null");
    Builder builder = new Builder(authorizationRequest.getAuthorizationUri());
    builder.clientId(authorizationRequest.getClientId());
    ...
    builder.state(authorizationRequest.getState());
    return builder;
}
```

# 전체 흐름 정리 (OAuth2)

> “Spring Security의 기본 OAuth 요청 생성기(DefaultOAuth2AuthorizationRequestResolver)를 감싸서,
커스텀 state를 추가한 새 요청(OAuth2AuthorizationRequest 불변객체)을 반환하는 구조.”

```
┌──────────────────────────────────────────┐
│  CustomOAuth2AuthorizationRequestResolver │
└──────────────────────────────────────────┘
                 │
                 ▼
    ┌────────────────────────────────┐
    │  DefaultOAuth2AuthorizationRequestResolver │ ← (Spring 기본 구현)
    └────────────────────────────────┘
                 │
                 ▼
     resolve(request, clientId)  ←  기본 인가 요청 생성
                 │
                 ▼
     customizeState(요청, HttpServletRequest)
         └─> redirectUrl, state 값 커스텀
                 │
                 ▼
     새로 build()된 OAuth2AuthorizationRequest 반환
```

✔️ defaultResolver는 스프링이 제공하는 기본 인가 요청 생성기
✔️ 우리는 그 결과(OAuth2AuthorizationRequest)를 받아서 state를 추가로 인코딩한 뒤 반환하는 것

# defaultResolver의 역할

```kotlin

private val defaultResolver = DefaultOAuth2AuthorizationRequestResolver(
    clientRegistrationRepository,
    OAuth2AuthorizationRequestRedirectFilter.DEFAULT_AUTHORIZATION_REQUEST_BASE_URI
)
```

-	clientRegistrationRepository:
스프링이 관리하는 OAuth2 클라이언트 설정 저장소
(예: application.yml에 있는 spring.security.oauth2.client.registration.kakao, google 등)
-	DEFAULT_AUTHORIZATION_REQUEST_BASE_URI:
/oauth2/authorization 이 기본 URI </br>
→ 예를 들어 /oauth2/authorization/kakao로 접속하면 카카오 로그인으로 리다이렉트하도록 설정됨.

즉, defaultResolver는
“요청을 보고, 어떤 클라이언트(kakao/google/naver)인지 판단해서
그에 맞는 OAuth2AuthorizationRequest를 만들어주는 기본 로직”을 담당함

## 두 개의 resolve() 함수의 차이

여기서 핵심은 “오버로딩된 메서드 (Overloaded Method)”
Spring Security의 OAuth2AuthorizationRequestResolver 인터페이스가 두 가지 시그니처를 요구하기 때문임 👇

```java
OAuth2AuthorizationRequest resolve(HttpServletRequest request);

OAuth2AuthorizationRequest resolve(HttpServletRequest request, String clientRegistrationId);
```

✅ (1) resolve(request: HttpServletRequest)
-	사용자가 URL로 접근할 때,
예를 들어 /oauth2/authorization/kakao 이런 기본 URI로 접근하면
내부적으로 clientRegistrationId (kakao)를 URI에서 추출해서 자동으로 넘겨줌.

즉, 이 버전은 “URI에서 clientId를 알아내는 일반 케이스”야.
(대부분 이게 호출됌)

⸻

✅ (2) resolve(request: HttpServletRequest, clientRegistrationId: String?)
-	OAuth 클라이언트를 프로그래밍적으로 직접 지정할 때 호출됨.

예시:
로그인 페이지에 버튼이 여러 개 있어서, JS에서
“카카오 로그인” 버튼 클릭 → /custom/oauth2/kakao 요청을 보낼 때
직접 clientRegistrationId = "kakao"를 명시할 수도 있지.

Spring Security 내부적으로도 OAuth2AuthorizationRequestRedirectFilter가
클라이언트를 명시적으로 찾을 때 이 오버로드를 사용해.

즉,

첫 번째 resolve()는 URI 기반 자동 추론
두 번째 resolve()는 명시적 clientRegistrationId 기반
